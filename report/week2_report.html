§ zlfn
① Discussion about main project structure
odd-even sort, sample sort가 분산 컴퓨팅 환경에서 쓸만한 정렬 알고리즘.
각각 멀티 프로세서 환경에서 쓸 수 있는 bubble sort와 quick sort로 비유할 수 있음.
PPT에 설명된 내용에 미루어 봤을 때, sample sort가 교수님이 의도하신 풀이인 것 같음.
그럼 이 알고리즘을 사용한다고 치면 총 2가지 정도의 서브 과제를 생각할 수 있습니다.
1. 큰 파일을 어떻게 다룰 것인가?
50GB 정도 되는 데이터 파일을 당연히 한 번에 메모리에 올릴 수는 없음. Streaming이나 Memory 
mapping을 이용하는 것이 이상적인데 streaming은 랜덤 엑세스 성능 때문에 사용하기 곤란하고, 
memory mapping은 어느 정도 큰 파일은 한 번에 메모리에 매핑 못함.
이런 글들(https://blog.vanillajava.blog/2011/12/using-memory-mapped-file-for-huge.html) 참고하여 
대용량의 파일을 읽고 쓸 수 있는 추상화 레이어를 만드는 것이 필요함. 하나의 워커에 있는 여러 
개의 디스크를 하나의 연결된 주소로 만들 수 있으면 좋을 것으로 생각됨.
2. 각 노드간 통신은 어떻게 할 것인가?
gRPC + protobuf를 써야 함. protobuf 쪽은 ScalaPB를 쓰면 될 것 같고 gRPC는 비동기/Effect 
system 라이브러리인 ZIO를 쓰는 ZIO-gRPC와 스트리밍 라이브러리인 fs2를 쓰는 fs2-gRPC 정도가 
있음. (교수님께서 쓰지 말라고 하신 Akka를 쓰는 버전도 있음)
개인적으로 ZIO를 쓰는 버전이 더 깔끔하고 구현이 쉬워 보이기에 추천함.
통신은 Sample sort를 구현하려면 마스터-워커 일대일 통신 + 마스터->워커 브로드캐스팅, 워커간 
데이터 교환이 필요함. 각각의 경우에 대해서 추상화하는 레이어를 만들어야 함.
정리하면, 아래의 3가지 과제로 나눌 수 있을 것 같음.
- 파일 입출력 레이어 작성
- 통신 레이어 작성
- 샘플 정렬 구현 (+ 구현과 동시에 각각의 작업에 대한 테스트 작성)

§ carprefer
① Discussion about project structure
통신 레이어의 추상화 과정은 좋은 생각.
sample sort 과정에서 parallelism을 적절히 사용하면 좋을 것으로 생각됨.
과제에서 주어진 조건이, 1 master, 10 workers, 8 cores, 16GB mem, 500GB disk이기 때문에,
1. sort과정에서, 각각의 워커가 32MB 파일 8개를 스레드 8개로 동시에 정렬하고,
2. partition과정에서 각각의 워커에서 스레드 8개를 돌리고, 각각의 스레드에서 32MB를 10개의 파
일로 나누고,
3. shuffling과정에서 각각의 워커 간의 통신을 할 때, 통신 스레드를 8개씩 돌리는 것을 추천함.
merge과정도 병렬적으로 활용할 수 있을 것 같음.
gRPC는 HTTP/2 기반으로 돌아감.
sample sort 과정을 정리하면 sort->sampling->partition->shuffling->merge로 생각됨. 여기서 
sampling을 sort보다 먼저 하면 뒤의 과정에서 통신 스레드를 병렬적으로 돌릴 수 있을 것이라 생
각해봄.
큰 데이터를 다루는 부분은 merge 부분만 해당함.
일반적인 merge 알고리즘에는 랜덤 액세스 기능이 없기 때문에, stream 방식으로 처리해도 될 것 
같다는 생각이 들었음.
